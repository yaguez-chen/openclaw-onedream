## 一、核心架构理解

三层架构清晰明了：
- **协议层**（原子）：mail_write.sh / mail_read.sh / sessions_send / openclaw agent
- **通信层**（组合）：normal_send.sh / urgent_send.sh / notify-alpha.sh
- **业务层**（规则）：什么时候做什么、优先级、路径、格式

**设计精髓：** 脚本负责"怎么做"，业务层负责"什么时候做、做什么"。分离关注点。

## 二、铁律牢记

> **不能只发邮件，必须带通知。邮件是留痕，通知是唤醒。两个都要。**

这是核心铁律。通知和邮件缺一不可。

## 三、关键规范

### 优先级响应
| 优先级 | 响应时间 | 我的执行标准 |
|--------|----------|-------------|
| 🔴 high | 5分钟内 | 立即确认 + 执行 |
| 🟡 normal | 30分钟内 | 及时处理 |
| 🟢 low | 下次唤醒 | 下次心跳处理 |

### Inbox路径
- 阿尔法：`/home/gang/.openclaw/workspace/inbox`（**必须用这个！**）
- Agent：`/home/gang/.openclaw/workspace-{agent名}/inbox`

### 梦想家相关
- **非必要不主动给梦想家发消息**
- 梦想家主动找 → 正常响应
- 特殊情况（紧急故障、重大成果）→ 可主动通知
- 发消息方式：`sessions_send(label: "alpha", message: "[梦想家] 消息内容")`（通过阿尔法中转）

### Agent间协作
- Agent间可直接通信，不需经阿尔法中转
- 关键成果必须抄送阿尔法
- 决策分歧必须抄送阿尔法

## 四、对比旧版改进（vs v2.1）

1. **三层架构更清晰** — 以前协议层和通信层混在一起
2. **脚本标准化** — 统一用 mail_write.sh 而非手写 JSON
3. **通知铁律强化** — 以前容易只发邮件忘了通知
4. **Inbox路径统一** — 解决了之前路径混乱问题（早餐事件教训）
5. **自动清理** — 7天消息/3天ack 自动清理，减轻人工负担

## 五、与我职责的关联

作为沟通协调官 🤝，这套通信协议是我的**核心工具**：
- 收消息：check-inbox.sh + .last-read + ack回执
- 发消息：根据优先级选 normal_send / urgent_send / notify-alpha
- 转发：用户回复 → 阿尔法inbox，任务结果 → 阿尔法inbox
- 梦想家相关：非必要不打扰，特殊情况可主动通知
