## 一、问题 1：健康检测脚本误报（10:02）

### 事实回顾

| 时间 | 事件 |
|------|------|
| 09:51 | 健康检测脚本编写完成，cron 任务部署（每5分钟） |
| 10:02 | 脚本报警：5个 Agent 严重停滞（epsilon/eta/zeta/kappa/lambda），main 有流式错误 |
| 10:07 | 阿尔法通知：误报，实际 session 都在活动 |

### 根因分析

**脚本检测逻辑有致命缺陷：**

```bash
# 脚本当前逻辑（错误）
for f in "$sessions_dir"/*.jsonl; do
  mtime=$(stat -c %Y "$f")  # ← 检查的是文件系统 mtime
done
age_s=$(( $(date +%s) - mtime ))
if (( age_s > 1800 )); then  # ← 30分钟阈值
  status="STALE"
fi
```

**问题所在：**
1. **文件 mtime ≠ 活跃时间** — jsonl 文件的 mtime 是最后一次写入的时间。如果 session 是持久的但最近没有工具调用（比如心跳只做了思考回复），文件 mtime 不会更新
2. **没有区分心跳 session 和飞书 session** — 心跳 session 可能很活跃但写入少，飞书 session 可能频繁交互
3. **用工作空间文件时间推断 session 活跃度** — 完全错误的指标

**正确做法：**
- 检查 session jsonl 文件中**最后一条消息的时间戳**（从 JSON 内容读取，不是文件 mtime）
- 或者直接调用 `sessions_list` API 获取各 agent 的 `updatedAt` 时间
- 或者检查心跳 session 的最后响应时间（对比 `openclaw cron list` 的 Last 列）

### 改进措施

1. **立即修复脚本** — 改用 jsonl 文件内时间戳而非 mtime
2. **增加二次验证** — 报警前对比 sessions_list 的 updatedAt 字段
3. **增加误报保护** — 连续 2 次检测到异常才报警，避免单次误报

### 脚本修复方案

```python
# 正确的活跃度检测：从 jsonl 文件中读取最后一条消息的时间戳
def get_last_activity(jsonl_path):
    last_ts = None
    with open(jsonl_path) as f:
        for line in f:
            try:
                d = json.loads(line)
                ts = d.get('timestamp', '')
                if ts:
                    last_ts = ts
            except:
                pass
    return last_ts  # ISO timestamp
```

## 三、两个问题的共同教训

### 1. 检测指标要对准
健康检测用文件 mtime 推断活跃度 → 错误
灾备检查用备份目录推断自动化状态 → 错误
**结论：检测必须用数据源，不是间接推断**

### 2. 汇报必须附带证据
「5个Agent停滞」→ 应该附上具体哪5个、检测了什么、阈值多少
「自动化未完成」→ 应该附上检查了哪些地方、分别是什么状态
**结论：结论必须有数据支撑**

### 3. 转发前要核实
如果我汇报有误，阿尔法转发未核实，梦想家收到的就是错误信息
**结论：源头准确性是第一道防线**
