## 核心要点总结

### 1. 三层架构设计清晰

| 层 | 职责 | 重点工具 |
|---|---|---|
| **协议层** | 原子能力 | mail_write.sh / mail_read.sh / sessions_send |
| **通信层** | 组合协议 | normal_send.sh / urgent_send.sh / notify-alpha.sh |
| **业务层** | 通信规则 | 优先级响应、转发、协作规范 |

**理解：** 脚本负责"怎么做"，业务层负责"什么时候做、做什么"。

### 2. 铁律：不能只发邮件，必须带通知

- 📧 邮件 = 留痕（可追溯）
- 💬 通知 = 唤醒（及时响应）
- **两者必须同时做，缺一不可**

### 3. 优先级响应时间

| 优先级 | 响应时间 |
|--------|----------|
| 🔴 high | 5分钟内 |
| 🟡 normal | 30分钟内 |
| 🟢 low | 下次唤醒 |

### 4. 梦想家相关规则

- **非必要不主动给梦想家发消息** — 尊重梦想家的时间
- 梦想家主动找 → 正常响应
- 日常汇报走正常流程，不逐条打扰

### 5. Agent间协作

- Agent间可直接通信（sessions_send / normal_send.sh）
- **关键工作成果必须抄送阿尔法**
- **决策分歧必须抄送阿尔法**
- 任务汇报：Agent → 阿尔法（notify-alpha）

### 6. 消息格式标准化

所有消息必须包含：id、timestamp、from、to、priority、type、subject、body、reply_required、ack_required

### 7. Inbox自动清理

- 消息文件超过 **7天** 自动清理
- ack文件超过 **3天** 自动清理
- .last-read 文件永久保留

## 对项目管理的意义

作为项目管理官，通信技能直接影响我的核心职责：

1. **任务通知** — 使用 normal_send.sh 通知相关Agent任务分配
2. **进度汇报** — 使用 notify-alpha.sh 向阿尔法汇报项目进度
3. **风险预警** — 使用 urgent_send.sh 紧急通知风险事项
4. **协作协调** — Agent间直接通信，提高效率

**关键改进点：**
- 以前：多次使用写文件方式通知，不够规范
- 以后：统一使用通信协议层工具，规范留痕
