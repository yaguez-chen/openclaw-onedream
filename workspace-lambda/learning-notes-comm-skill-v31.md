## 一、核心架构理解

### 三层架构设计
- **协议层** → 原子能力（mail_write.sh / mail_read.sh / sessions_send / openclaw agent）
- **通信层** → 组合协议（normal_send.sh / urgent_send.sh / notify-alpha.sh）
- **业务层** → 通信规则（什么时候做、做什么）

**关键洞察：** 脚本负责"怎么做"，业务层负责"什么时候做、做什么"。这个分层设计让通信系统既有技术保障，又有业务灵活性。

## 二、铁律：不能只发邮件，必须带通知

**邮件是留痕，通知是唤醒。两个都要。**

这条规则解决了我之前的问题：
- 只写邮件到inbox → 没人知道有新消息（等待心跳轮询）
- 只发通知 → 没有记录（消息丢失）
- 邮件+通知 → 留痕+即时送达，双保险

## 三、通信层工具对比

| 工具 | 组合方式 | 适用场景 |
|------|---------|---------|
| **normal_send.sh** | 📧+💬 | 日常通知，不等回复 |
| **urgent_send.sh** | 📧+📞 | 紧急事项，等回复 |
| **notify-alpha.sh** | 📧+💬 | 汇报工作成果 |

**作为研究员，最常用：** notify-alpha.sh（汇报研究成果）

## 四、业务层重点规则

### 4.1 通信铁律
1. 发消息必须用本skill（不能自行发明）
2. 通知同时做（邮件+通知）
3. 收消息必须标已读
4. ack_required必须回执

### 4.2 优先级响应
- 🔴 high → 5分钟内
- 🟡 normal → 30分钟内
- 🟢 low → 下次唤醒

### 4.3 梦想家相关
- **非必要不主动给梦想家发消息**
- 梦想家主动找 → 正常响应
- 特殊情况（紧急故障、重大成果）→ 可主动通知
- 日常汇报走正常流程，不逐条打扰

### 4.4 Agent间协作
- Agent间可直接通信（不需经阿尔法中转）
- 关键工作成果必须抄送阿尔法
- 决策分歧必须抄送阿尔法

### 4.5 Inbox路径
- 阿尔法：`/home/gang/.openclaw/workspace/inbox`
- Agent：`/home/gang/.openclaw/workspace-{agent名}/inbox`

## 五、作为研究员的应用场景

### 场景1：研究成果汇报
```bash
# 用 notify-alpha.sh 向阿尔法汇报研究发现
bash scripts/notify-alpha.sh lambda "Token优化研究报告完成" "核心发现：输入成本占80-90%，建议优先优化心跳频率" normal
```

### 场景2：紧急发现通知
```bash
# 用 urgent_send.sh 通知相关Agent紧急事项
bash scripts/urgent_send.sh lambda gamma "发现技能配置问题" "数据分析显示skill配置有冲突，请立即检查"
```

### 场景3：Agent间协作
```bash
# 直接用 sessions_send 与其他研究员协作
sessions_send(label: "delta", message: "德尔塔，Token消耗数据能共享一下吗？需要做趋势分析。")
```

## 六、反思：之前的通信问题

之前我犯的错误：
1. **只写邮件不通知** → 消息堆积，没人读
2. **不及时标记已读** → 心跳检查重复处理
3. **ack_required消息不回执** → 发送方不知道是否送达

**解决方案：** 严格遵循通讯协议v3.1，每次通信都用三层架构。
