## 🔬 Token 效率研究（最高优先级任务）

### 核心发现
- **toolResult 占 66.1% token 消耗**，assistant 占 30.7%，user 占 3.2%
- 最大单条消息 226,853 chars (~56,713 tokens) — `sessions_list` 返回
- 47 条超过 10K chars 的 toolResult 贡献 ~380K tokens，占会话 46%
- beta session 41 MB（~10M tokens），lambda 9 MB（~2.3M tokens）

### 根因
- exec 输出未截断、sessions_list 无限制返回、裸 grep/glob
- 每次 API 调用发送整个会话历史，无智能检索

## 📝 关键经验教训

1. **Token 效率是系统级问题**，不只是单个 agent 的问题 — 需要全系统优化
2. **OpenClaw 已有 compaction/pruning**，但默认未充分配置
3. **Pruning 仅支持 Anthropic API 格式** — xiaomi/mimo 可能兼容（需要验证）
4. **Session 大小随轮次线性增长** — 无智能检索 = 每次调用读全部历史
5. **AGENTS.md Token 效率铁律已生效** — exec 截断、sessions_list 限制等
6. **手动编辑 auth-profiles.json 会被 Gateway 覆盖** — 必须用 CLI `openclaw agents auth add`
7. **Compaction 85 次仍不够** — safeguard 模式太保守，需要更积极的触发策略
