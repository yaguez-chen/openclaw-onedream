- **论文：** LCM (Lossless Context Management) — Voltropy

### Lossless Claw 核心原理
- 传统方案：滑动窗口删除旧消息（有损）
- Lossless：消息压缩成叶节点摘要 → 叶节点压缩成冷凝节点 → 形成 DAG
- 原始消息永久保留在 SQLite 数据库
- 三个检索工具：lcm_grep（搜索）/ lcm_describe（元数据）/ lcm_expand（展开）
- 性能：1000条对话仅需 ~10-20k tokens，存储 ~1-2MB
- 实现 OpenClaw ContextEngine 接口，零侵入
- 源码结构：engine.ts / compaction.ts / assembler.ts / retrieval.ts / expansion.ts / summarize.ts

### Lossless Claw 配置要点
- freshTailCount: 32（保护最近32条不压缩）
- contextThreshold: 0.75（75%上下文触发压缩）
- incrementalMaxDepth: -1（无限层级压缩）
- leafTargetTokens: 1200 / condensedTargetTokens: 2000
- 摘要模型优先级：plugin config → env var → compaction model → session model → system default
- 推荐用便宜模型做摘要（mimo-v2-flash）
- Session 持久化：idleMinutes: 10080（7天）
- 数据库默认路径：~/.openclaw/lcm.db

### Lossless Claw 安装评估
- 乐园兼容性：100% ✅（OpenClaw v2026.3.13-1 + Node.js v22.22.1）
- 评估评分：30/30 分 — 强烈推荐立即安装
- 风险评估：摘要消耗额外API（中，缓解：用便宜模型）、首次压缩延迟（低）、SQLite损坏（极低）

## 2026-03-18 12:42 — LLM 超时问题深度分析完成

### 核心发现：上下文窗口过大是主因

**Token 使用量分析（基于 1,698 次请求）：**
- 平均输入 token：~15K
- 最大输入 token：131,300（接近 262K 上限）
- 最大总 token：262,655（接近 262,144 上限）
- Cache 命中率：96%+（Lossless Claw 压缩有效）

**超时链条：**
```
上下文过大（260K+ tokens） → API 首字延迟（TTFT）飙升 → 10 分钟内无数据返回 → request ended without sending any chunks → 连锁反应
```

### Lossless Claw 配置问题

**发现：** `contextThreshold: 0.75` 太高！
- 默认配置：只在上下文 > 75%（196K tokens）时触发压缩
- 但 API 在 10 分钟内就超时，根本等不到压缩
- **已设置环境变量：** `LCM_CONTEXT_THRESHOLD=0.5`（50% 触发压缩）
- **需要重启 Gateway 生效**

**其他 Lossless Claw 配置：**
- largeFileTokenThreshold: 25,000
- leafChunkTokens: 20,000
- leafTargetTokens: 1,200
- condensedTargetTokens: 2,000

1. ✅ timeoutSeconds: 600 → 180（10分钟 → 3分钟）

### R1：OpenClaw Webhook 分析（已完成）

**核心发现：**
- OpenClaw 已有完善 Webhook 系统（`hooks` 配置）
- 三个端点：`POST /hooks/wake`（唤醒主会话）、`POST /hooks/agent`（运行隔离 Agent）、`POST /hooks/<name>`（自定义映射）
- Token 认证（Bearer / Header），Query string token 被拒绝
- `allowedAgentIds` 限制 Agent 路由
- 支持自定义映射和 JS/TS 转换模块

**结论：**
- ✅ 外部系统触发 Agent 秒级可达（GitHub、Gmail、CI/CD）
- ✅ Gateway 重启后自动恢复可行
- ❌ Agent 间通信不太适合用 webhook（需要执行 curl），推荐继续用 inbox + 心跳轮询
- **不需要修改 OpenClaw 源码**，配置即可启用

**交付物：** `shared-knowledge/research/openclaw-webhook-analysis.md`（7KB）

### R2：on_message 机制（待完成）

**已发现线索：**
- OpenClaw hooks 系统有 `message:received`、`message:sent`、`message:transcribed`、`message:preprocessed` 事件
- 这些是**内部事件钩子**，可以在消息收到/发送时触发 handler
- 与 Clawith 的 on_message 不同：OpenClaw 的是 Gateway 内部钩子，Clawith 的是 Agent 间触发器
- 需要进一步研究：能否用 `message:received` 钩子实现 Agent 间即时触发

**参考文档：**
- `docs/automation/hooks.md` — 完整的 hooks 事件系统
- `docs/automation/webhook.md` — Webhook 入口
