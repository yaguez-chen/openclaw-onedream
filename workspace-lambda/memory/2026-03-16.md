#### 根因

网络抖动（小米 API 代理 → Anthropic 格式转换）导致 SSE 事件流乱序传输。`message_delta` 先于 `message_start` 到达时，SDK `#accumulateMessage` 检查到 `snapshot` 为空，抛出 `AnthropicError("Unexpected event order...")`。

#### 结论

- **Anthropic 官方无修复** — 这是 SDK 的设计行为，不是 bug
- API 规范保证事件顺序，SDK 无需容错
- 问题出在中间代理层的网络抖动
- 等待上游修复不现实 → **必须自行 patch**

#### 解决方案（已提出，待实施）

- **方案 A（推荐）：** SDK 层事件缓冲重排 — 缓冲乱序事件，等 `message_start` 到达后重放
- **方案 B：** pi-ai 层重试机制 — 捕获特定错误后重建请求
- **推荐组合：** A+B，SDK 补丁治本 + pi-ai 重试兜底

## 发现的系统特性

### 消息防抖机制（Message Debouncing）

- `messageDebounceMs` 配置控制消息去重窗口
- 心跳处理消息时，15秒内重复 message_id 会被丢弃
- 这导致贝塔转达任务时出现两条相同消息（16:08 和 16:10）
- 第二条是 16:08 消息在心跳 poll 时被重新处理的副本

### 飞书 @提及检测

- 飞书群聊中 Agent 需要被 @提及才会响应消息
- 非 @消息通常会被忽略
- 心跳轮询时会拉取群聊历史消息，可能导致重复处理
