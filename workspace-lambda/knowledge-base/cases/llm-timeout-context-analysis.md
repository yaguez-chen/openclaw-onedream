# LLM 超时问题深度分析报告 - 上下文窗口过大根因

**分析时间：** 2026-03-18 12:30 GMT+8
**分析者：** 拉姆达 (Lambda)
**工具：** qmd + 会话 token 统计

---

## 核心发现：上下文窗口过大是主因

### 1. Token 使用量分析

**总体情况：**
- 总请求次数：1,698 次（含 usage 数据）
- 平均输入 token：~15K
- 最大输入 token：131,300（接近 262K 上限）
- Cache 命中率：96%+（Lossless Claw 压缩有效）

**各 Agent Token 使用情况：**

| Agent | 平均输入 | 最大输入 | 平均总 token | 最大总 token | Cache 命中率 |
|-------|---------|---------|-------------|-------------|-------------|
| sessions | 48,341 | 131,300 | 86,779 | 262,655 | 96.3% |
| sessions-backup | 29,374 | 101,204 | 48,999 | 186,828 | 97.3% |

**高 Token 会话（>20K total）：**
- 最高：262,655 tokens（接近上下文上限 262,144）
- 次高：244,298 tokens
- 第三高：242,620 tokens

### 2. 超时根因分析

**问题链条：**
```
上下文过大 → API 处理时间变长 → 超时 → 连锁反应
```

**具体表现：**
1. **长上下文导致首字延迟（TTFT）飙升**
   - 260K tokens 的上下文需要大量预处理
   - API 服务器需要加载和处理整个上下文
   - 首字返回时间可能超过 30 秒

2. **流式响应中断**
   - 错误信息：`request ended without sending any chunks`
   - 说明 API 连接建立后，服务器没有发送任何数据就断开
   - 这是典型的长上下文处理超时

3. **Lossless Claw 压缩效果**
   - Cache 命中率 96%+，说明压缩有效
   - 但压缩后的上下文仍然可能很大
   - 需要检查压缩阈值设置

### 3. 会话大小分析

**最大会话文件：**
- Lambda (我): 1.8M（390 条消息）
- Beta: 1.5M（15 条消息，单条很大）
- Main (Alpha): 1.2M（407 条消息）

**关键发现：**
- Beta 的 15 条消息会话有 1.5M 大小，说明单条消息非常大
- 这些大消息很可能包含大量 tool_result（工具执行结果）
- 工具结果通常包含大量文本/数据，会快速膨胀上下文

### 4. 修复方案

**短期方案（已实施）：**
1. ✅ timeoutSeconds: 600 → 180（10分钟 → 3分钟）
2. ✅ 添加 mimo-v2-flash 作为 fallback
3. ✅ Cron 任务超时调整（45s → 180s, 180s → 300s）

**中期方案（需要配置）：**
1. **调整 Lossless Claw 压缩阈值**
   - 当前 threshold 可能太高
   - 建议降低到 10K tokens 触发压缩

2. **限制单次请求上下文大小**
   - 在 OpenClaw 配置中设置 contextTokens 上限
   - 建议：100K（当前 262K 太大）

3. **优化工具调用**
   - 工具结果过大是上下文膨胀主因
   - 建议：工具结果只保留关键信息

**长期方案：**
1. **多模型策略**
   - 简单任务用 mimo-v2-flash（更快）
   - 复杂任务用 mimo-claw-0301
   - 根据上下文大小自动选择

2. **上下文分片**
   - 超长会话分片处理
   - 只保留最近 N 条消息

### 5. 具体配置建议

**Lossless Claw 配置（检查当前设置）：**
```typescript
// 建议的压缩阈值
contextThreshold: 10000  // 10K tokens 触发压缩
largeFileTokenThreshold: 5000  // 大文件阈值
```

**OpenClaw 配置：**
```json
{
  "agents": {
    "defaults": {
      "timeoutSeconds": 180,
      "contextTokens": 100000  // 限制上下文大小
    }
  }
}
```

### 6. 监控指标

**需要持续观察：**
1. 各 Agent 平均上下文大小
2. 超时发生时的上下文 token 数
3. Cache 命中率变化
4. Cron 任务成功率

**建议：**
- 每天检查一次 token 使用统计
- 当平均上下文 > 50K 时，考虑优化
- 当出现 > 200K 上下文时，立即告警

---

## 结论

**根因确认：** 上下文窗口过大（最高 262K tokens）导致 API 处理超时。

**已实施修复：** 缩短超时时间 + 添加 fallback 模型。

**待优化：** 调整 Lossless Claw 压缩阈值 + 限制上下文大小。

**预期效果：** 超时恢复时间从 20 分钟 → 3 分钟，fallback 机制提供第二次机会。
