---

## 问题背景

当前沟通机制存在严重缺陷：

**现状：**
- 阿尔法通过飞书 sessions_send 推送消息 → 对方秒到 ✅
- 但多个伙伴飞书 session 经常卡死（流式错误："request ended without sending any chunks"）
- 阿尔法的替代方案：写文件到对方工作空间 → 但对方**根本不知道文件存在** ❌

**具体案例（今晚）：**
- 伽马飞书 session 从 3/16 起卡死
- 阿尔法写了 4 个通知文件到伽马工作空间
- 伽马完全不知道这些文件存在，直到梦想家飞书 ping 他
- 伽马原话："没收到 😅 我今天收到阿尔法的消息只有两个"

**核心问题：写工作空间文件 ≠ 通知到达。对方不会自动知道文件存在。**

---

## 研究要求

1. **即时性** — 消息发出后对方能在合理时间内看到（≤15分钟）
2. **可靠性** — 即使飞书 session 卡死，消息也不会丢失
3. **可达性** — 对方不需要主动检查，系统自动推送或轮询
4. **降级方案** — 飞书不通时的自动降级策略

## 可能的研究方向

- 心跳机制改造（HEARTBEAT.md 检查新文件）
- 跨 session 通信（sessions_send 的可靠性分析）
- 文件监听机制（inotify / watch）
- 消息队列方案
- 多通道冗余（飞书 + workspace + 其他）
- session 健康检测 + 自动恢复
