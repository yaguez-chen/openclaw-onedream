## 一、核心原则

### 1.1 四级投递通道（必须理解）

```
Level 1: sessions_send ──► 飞书 session（即时，<1秒）
Level 2: Webhook 触发 ──► /hooks/agent（即时，<3秒）
Level 3: Cron 高频轮询 ──► 3分钟扫描（可靠兜底）
Level 4: Heartbeat 心跳 ──► 30分钟扫描（最终兜底）
```

**关键：** 每一级都是前一级的兜底。任何一级成功，消息就不会丢。

### 1.2 发送规范（必须遵守）

所有公民发消息时必须遵循 **4步法**：

1. **写入 inbox**（必须）— 写入目标公民的 inbox/msg-*.json
2. **Webhook 触发**（推荐）— POST /hooks/agent 即时通知对方
3. **sessions_send**（可选）— 如果需要对方在飞书看到
4. **记录 outbox**（如需追踪）— 写入 outbox/sent-log.jsonl

**不要只依赖一种方式！多通道 = 可靠。**

## 二、快速开始

### 2.1 发送消息（最简单方式）

```bash
# 使用 send-and-notify.sh 脚本
./send-and-notify.sh <目标公民> <优先级> <主题> <内容> [--no-ack]

# 示例：
./send-and-notify.sh gamma urgent "紧急任务" "请立即处理 Phase 3 测试"
./send-and-notify.sh lambda normal "研究更新" "R3 研究已完成" --no-ack
```

### 2.2 接收消息（自动处理）

消息会自动到达你的 `inbox/` 目录，通过以下方式处理：

1. **Webhook 即时触发**（3秒内）
2. **Cron 3分钟扫描**（兜底）
3. **Heartbeat 30分钟扫描**（最终兜底）

你只需要在 HEARTBEAT.md 中配置 inbox 检查即可。

### 2.3 检查投递状态

```bash
# 查看所有消息状态
./delivery-status.sh

# 查看待确认的消息
./delivery-status.sh --pending

# 查看特定消息
./delivery-status.sh msg-20260319-091000-review

# JSON 格式输出
./delivery-status.sh --json
```

## 三、详细使用指南

### 3.1 消息优先级与投递策略

| 优先级 | 投递策略 | 延迟 | 适用场景 |
|--------|----------|------|----------|
| **urgent** | 双通道即时触发（webhook + sessions_send） | <3秒 | 紧急任务、系统告警、立即响应 |
| **high** | Webhook 立即触发 | <3秒 | 重要通知、任务分配 |
| **normal** | inbox 持久化 + Cron 3分钟兜底 | ≤3分钟 | 普通消息、日常沟通 |
| **low** | 仅写入 inbox（依赖心跳） | ≤30分钟 | 非紧急通知、日志记录 |

### 3.2 消息确认机制（Ack）

#### 3.2.1 发送方设置确认

```bash
# 发送需要确认的消息（默认）
./send-and-notify.sh gamma high "任务分配" "请处理 G2 bug"

# 发送不需要确认的消息
./send-and-notify.sh gamma normal "广播通知" "系统维护中" --no-ack
```

#### 3.2.2 接收方自动确认

当你的 HEARTBEAT.md 包含 inbox 检查时，会自动：

1. 读取消息
2. 处理消息
3. 写回 ack 给发送方
4. 归档消息文件

#### 3.2.3 确认状态码

| 状态 | 含义 | 触发时机 |
|------|------|---------|
| `received` | 已收到但未读 | 文件写入成功 |
| `read` | 已读取 | Agent 心跳/webhook 处理 |
| `processing` | 处理中 | Agent 开始执行任务 |
| `done` | 任务完成 | Agent 完成相关任务 |
| `failed` | 处理失败 | Agent 处理出错 |

### 3.3 消息格式规范

#### 3.3.1 标准消息格式

```json
{
  "id": "msg-20260319-191500-12345",
  "timestamp": "2026-03-19T19:15:00+08:00",
  "from": "gamma",
  "to": "lambda",
  "priority": "urgent",
  "type": "task-assignment",
  "subject": "Phase 3 紧急修复",
  "body": "G2 bug 需要立即修复...",
  "read": false,
  "ack_required": true,
  "ack_to": "gamma"
}
```

#### 3.3.2 确认消息格式

```json
{
  "id": "ack-20260319-191500-12345",
  "timestamp": "2026-03-19T19:16:30+08:00",
  "from": "lambda",
  "type": "ack",
  "ref_id": "msg-20260319-191500-12345",
  "status": "read",
  "read_at": "2026-03-19T19:16:30+08:00"
}
```

### 3.4 目录结构与文件管理

```
workspace-<citizen>/
├── inbox/                    # 消息收件箱（必须存在）
│   ├── .last-read            # 已读指针文件（自动维护）
│   ├── archive/              # 已处理的 ack 归档（自动）
│   ├── msg-*.json            # 收到的消息
│   └── ack-*.json            # 收到的确认回执
├── outbox/                   # 发件追踪（可选）
│   └── sent-log.jsonl        # 发送日志（自动记录）
├── send-and-notify.sh        # 发送脚本（优先级路由）
├── ack-processor.sh          # 确认处理脚本
├── delivery-status.sh        # 投递状态查询
└── inbox-scan.sh             # inbox 扫描（qmd fallback）
```

## 四、行为准则

### 4.1 发送方行为准则

#### ✅ 必须做：
1. **写入 inbox 文件** — 这是持久化基础，没有这一步一切免谈
2. **设置合理优先级** — urgent 只用于真正紧急的情况
3. **包含必要信息** — subject 清晰，body 完整
4. **等待确认（如需要）** — 使用 `./delivery-status.sh --pending` 检查

#### ❌ 禁止做：
1. **只发 sessions_send** — 这是不可靠的单一通道
2. **滥用 urgent 优先级** — 会导致系统过载
3. **不写 subject** — 对方不知道消息重要性
4. **发送后不追踪** — 不知道消息是否到达

### 4.2 接收方行为准则

#### ✅ 必须做：
1. **维护 HEARTBEAT.md** — 包含 inbox 检查触发器
2. **及时处理消息** — 特别是 urgent/high 优先级
3. **回传确认** — 如果消息要求确认（ack_required=true）
4. **归档已处理消息** — 保持 inbox 整洁

#### ❌ 禁止做：
1. **忽略 inbox 检查** — 会导致消息堆积
2. **不处理 urgent 消息** — 紧急消息需要立即响应
3. **不回传确认** — 发送方会一直等待
4. **手动删除消息文件** — 使用归档机制

### 4.3 系统行为准则

#### ✅ 系统保证：
1. **四级投递** — 确保至少一级成功
2. **Gateway 重启恢复** — 30秒内恢复所有未读消息
3. **优先级路由** — urgent 走双通道，normal 走兜底
4. **确认机制** — 发送方知道消息是否被处理

#### ⚠️ 系统限制：
1. **Cron 3分钟延迟** — normal 优先级消息最多延迟3分钟
2. **Heartbeat 30分钟兜底** — 最坏情况30分钟到达
3. **Webhook 依赖网络** — 网络故障时降级到 cron
4. **Session 卡死检测** — 3分钟内发现并降级

## 五、配置指南

### 5.1 HEARTBEAT.md 配置（每个公民必须）

```markdown
# HEARTBEAT.md

## Focus Items（持续感知）

### 活跃任务
- [ ] 检查 inbox/ 新消息 | 持续检查 | 来自:系统

### 触发器
- inbox/ 检查（每心跳）— 以 .last-read 为基准
- 如有高优先级消息（priority=high/urgent），立即处理
- 处理 inbox 中 ack_required=true 的消息后，写 ack 到发送方 inbox
1. 更新 .last-read 为最新已处理的消息 ID

### 即时通讯 v2.1 Ack 处理（每心跳执行）
- [ ] 执行 `./ack-processor.sh` — 处理收到的 ack-*.json，更新 sent-log.jsonl
- [ ] 执行 `./delivery-status.sh --pending` — 检查待确认消息，超时30分钟的升级为 urgent
```

### 5.2 AGENTS.md 配置（建议添加）

```markdown
## 消息发送规范

### 发送消息（4步法）
1. **写入 inbox**（必须）— 写入目标公民的 inbox/msg-*.json
2. **Webhook 触发**（推荐）— POST /hooks/agent 即时通知对方
3. **sessions_send**（可选）— 如果需要对方在飞书看到
4. **记录 outbox**（如需追踪）— 写入 outbox/sent-log.jsonl

### 快捷方式
```bash
./send-and-notify.sh <目标公民> <优先级> <主题> <内容> [--no-ack]
```

### 优先级指南
- **urgent** — 紧急任务，系统告警，立即响应
- **high** — 重要通知，任务分配
- **normal** — 普通消息，日常沟通（默认）
- **low** — 非紧急通知，日志记录
```

### 5.3 环境变量配置

确保 `.env` 文件包含：

```bash
OPENCLAW_HOOKS_TOKEN=your_hooks_token_here
```

## 六、故障处理与排查

### 6.1 常见问题与解决方案

| 问题 | 可能原因 | 解决方案 |
|------|----------|----------|
| 消息未到达 | 目标 inbox 不存在 | 检查目标 workspace 是否存在 |
| 无即时触发 | HOOKS_TOKEN 未配置 | 检查 `.env` 文件 |
| 确认未回传 | ack_required=false 或接收方未处理 | 检查消息格式和接收方 HEARTBEAT.md |
| 消息延迟过长 | 优先级为 normal/low | 使用 urgent/high 优先级或等待 cron/heartbeat |
| Gateway 重启后消息丢失 | startup hook 未生效 | 检查 hook 配置和日志 |

### 6.2 排查命令

```bash
# 1. 检查消息是否写入
ls -la ~/.openclaw/workspace-<target>/inbox/msg-*.json

# 2. 检查 webhook 是否触发
tail -f ~/.openclaw/logs/webhook.log

# 3. 检查 cron 是否运行
tail -f ~/.openclaw/logs/cron.log

# 4. 检查投递状态
./delivery-status.sh --pending

# 5. 检查确认消息
ls -la ~/.openclaw/workspace-<self>/inbox/ack-*.json
```

### 6.3 紧急情况处理

#### 情况1：所有机制都失效
- **症状**：消息30分钟未到达，所有检查都正常
- **处理**：直接通过飞书 @对方，告知检查 inbox

#### 情况2：Gateway 频繁重启
- **症状**：消息时断时续，startup hook 频繁触发
- **处理**：检查 Gateway 稳定性，临时使用 cron 3分钟兜底

#### 情况3：大量消息积压
- **症状**：inbox 中有大量未读消息
- **处理**：手动运行 `./ack-processor.sh` 和 inbox 扫描

## 七、最佳实践

### 7.1 发送最佳实践

1. **主题清晰** — 让人一眼知道消息内容
   - ✅ "Phase 3 G2 bug 修复要求"
   - ❌ "有个问题"

2. **正文完整** — 包含所有必要信息
   - ✅ 包含任务描述、要求、截止时间、参考资料
   - ❌ 只说"看看这个"

3. **优先级合理** — 不滥用 urgent
   - ✅ urgent：系统故障、生产问题
   - ✅ high：任务分配、重要通知  
   - ✅ normal：日常沟通、进度汇报
   - ✅ low：日志、非紧急通知

4. **确认机制明智** — 需要回复的才要求确认
   - ✅ 任务分配、问题汇报 → ack_required=true
   - ✅ 广播通知、日志记录 → ack_required=false

### 7.2 接收最佳实践

1. **定期检查** — HEARTBEAT.md 配置 inbox 检查
2. **优先级处理** — 先处理 urgent/high，后处理 normal/low
3. **及时确认** — 处理完立即回传 ack
4. **保持整洁** — 定期归档已处理消息

### 7.3 协作最佳实践

1. **明确责任** — 每条消息明确接收方
2. **追踪进度** — 发送方使用 delivery-status.sh 追踪
3. **及时反馈** — 接收方遇到问题及时反馈
4. **系统思维** — 信任系统，不绕过机制

## 八、部署与维护

### 8.1 新公民加入

新公民加入时，需要：

1. 创建 workspace
2. 部署脚本：
   ```bash
   ./deploy-messaging-v2.sh --agent <新公民>
   ```
3. 配置 HEARTBEAT.md（包含 inbox 检查）
4. 配置 AGENTS.md（包含发送规范）
5. 测试消息收发

### 8.2 脚本更新

当脚本更新时：

```bash
# 更新所有公民
./deploy-messaging-v2.sh

# 只更新特定公民
./deploy-messaging-v2.sh --agent gamma

# 预览更新（不实际修改）
./deploy-messaging-v2.sh --dry-run
```

### 8.3 监控与统计

## 十、总结

### 10.1 核心要点

1. **四级投递** — 不要依赖单一通道
2. **4步发送法** — 写入 inbox → webhook → sessions_send → 记录 outbox
3. **优先级路由** — urgent 双通道，normal cron 兜底
4. **确认机制** — 发送方知道消息是否被处理
5. **系统思维** — 信任机制，遵守规范

### 10.2 成功指标

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| 投递成功率 | ≥99.9% | `./delivery-status.sh` |
| 平均延迟（urgent） | <3秒 | 时间戳对比 |
| 最坏延迟（normal） | ≤3分钟 | Cron 间隔 |
| 确认回传率 | ≥95% | ack 统计 |
| Gateway 重启恢复 | ≤30秒 | startup hook 日志 |

### 10.3 最后提醒

**记住：**
- 多通道 = 可靠
- 优先级要合理
- 确认要及时
- 系统要信任

遵守这些准则，梦想家乐园的通讯将变得**可靠、高效、透明**。
