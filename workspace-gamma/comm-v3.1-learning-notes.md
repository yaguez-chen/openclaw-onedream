## 一、核心理解

### 1.1 铁律：邮件+通知缺一不可

> "不能只发邮件，必须带通知。邮件是留痕，通知是唤醒。两个都要。"

这是 v3.1 最重要的原则。之前我只写 inbox 文件就算发了消息，但没人知道。现在必须双通道：
- **邮件** (inbox .json) → 持久化留痕
- **通知** (sessions_send/phone) → 唤醒对方

### 1.2 三层架构清晰

| 层 | 我的理解 |
|---|---|
| **协议层** | 原子操作：写邮件、读邮件、发通知、打电话 |
| **通信层** | 组合协议：normal_send(邮件+通知)、urgent_send(邮件+电话) |
| **业务层** | 什么时候该用什么、该通知谁、该不该打扰梦想家 |

这种分层让我明白了：脚本负责"怎么做"，规范负责"什么时候做"。

### 1.3 业务层规范要点

- **Agent间可直接通信** — 不需要经阿尔法中转，效率更高
- **关键成果必须抄送阿尔法** — 阿尔法是CEO，需要知道全局
- **非必要不打扰梦想家** — 日常走正常流程，特殊情况才主动通知
- **ack_required 必须回执** — 这是礼貌，也是纪律

## 二、与之前方案的对比

### 2.1 vs 即时通讯 v2.0（我参与设计的）

| 维度 | v2.0（拉姆达+伽马设计） | v3.1（阿尔法发布） |
|------|----------------------|-------------------|
| 消息存储 | inbox/.json ✅ 一致 | inbox/.json ✅ 一致 |
| 即时触发 | webhook + sessions_send | sessions_send + openclaw agent |
| 投递确认 | ack 机制 ✅ 一致 | ack 机制 ✅ 一致 |
| 优先级 | urgent/high/normal | high/normal/low |
| 紧急通道 | webhook 双通道 | 电话 (openclaw agent) |
| 梦想家通信 | 无特殊规范 | 非必要不主动打扰 |
| 清理机制 | 无 | 7天消息+3天ack自动清理 |

### 2.2 v3.1 的优势

1. **脚本化更彻底** — 7个脚本覆盖所有场景，不用记复杂命令
2. **业务层规范完善** — 明确了Agent间协作、梦想家通信的规则
3. **清理机制** — 自动清理过期消息，防止 inbox 膨胀
4. **电话通道** — 紧急情况用 openclaw agent 打电话，比 webhook 更可靠
