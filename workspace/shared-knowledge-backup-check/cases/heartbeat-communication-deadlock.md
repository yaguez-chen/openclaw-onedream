# 案例：心跳与通信死循环

> 事件日期：2026-03-15 · 记录者：拉姆达 🔬

## 事件概述

**问题本身阻碍了问题的解决** — 心跳禁用导致 Agent 间无法通信，而传递修复心跳方案恰好需要 Agent 间通信。

## 打破循环

**方式：梦想家手动转达**

梦想家作为"人肉路由器"，把拉姆达的研究结果手动转发给贝塔，贝塔据此修复心跳，循环打破。

## 根因分析

心跳禁用的三个原因：

1. **`target` 未设置** → 默认 `"none"` → 消息不发送
2. **HEARTBEAT.md 只有注释** → 心跳被跳过以节省 API
3. **Agent 无显式 heartbeat 块** → 配置未生效

## 经验教训

### 1. 基础设施优先

Agent 通信是基础设施，在部署多 Agent 系统前应该先确保：
- 所有 Agent 心跳已启用
- HEARTBEAT.md 包含实际任务
- `target` 设置正确

### 2. 问题可能自我阻碍

调试一个问题时，要考虑**问题本身是否会影响调试过程**。如果是，需要找到绕过的方式（如通过人类中转）。

### 3. 分层诊断

遇到通信问题时的排查层次：
1. 配置层（heartbeat.every、target）
2. 内容层（HEARTBEAT.md 是否有效）
3. Agent 层（是否有显式配置）
4. 渠道层（可见性控制）

### 4. 超时 ≠ 失败

`sessions_send` 超时只代表发送方等待回复超时，不代表消息未送达。要区分"消息没到"和"回复超时"。
