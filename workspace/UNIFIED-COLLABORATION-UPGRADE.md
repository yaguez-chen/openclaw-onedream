## 一、核心问题

乐园当前通信四大根因：
1. **飞书 session 不稳定** — 流式错误导致 session 卡死，消息入队但无法处理
2. **工作空间文件 ≠ 通知** — 写文件是磁盘操作，Agent 没有文件监听机制
3. **心跳形同虚设** — HEARTBEAT.md 为空，心跳被跳过
4. **无降级策略** — 飞书不通时没有自动切换

**结果：** 消息丢失、通知延迟数小时、Agent 协作效率低。

## 三、核心组件设计

### 3.1 三层消息传递（继承原方案）

| 层级 | 通道 | 延迟 | 可靠性 | 用途 |
|------|------|------|--------|------|
| 第一层 | sessions_send（飞书） | <1秒 | 依赖session | 即时通信 |
| 第二层 | workspace inbox/ 文件 | ≤5分钟 | 磁盘持久化 | 备选+降级 |
| 第三层 | HEARTBEAT.md 心跳轮询 | ≤30分钟 | 只要心跳运行 | 兜底 |

### 3.2 Aware 持续感知（Clawith 借鉴）

**核心思想：** HEARTBEAT.md 从"空文件"升级为"Focus Items 看板"。

**标准模板：**

```markdown
# HEARTBEAT.md

## Focus Items（持续感知）
### 活跃任务
- [ ] task-001: [任务描述] | 等待[某人/某事] | 来源:[谁]:发:[日期]
- [/] task-002: [任务描述] | 进行中 | 截止:[时间]
- [x] task-003: [任务描述] | 已完成 | [完成信息]

### 触发器
- inbox/ 检查（每心跳）
- 如有高优先级消息，立即处理
- 如 task-XXX 等待条件满足，更新状态并处理

### 待处理
- 检查 inbox/ 目录新消息（以 .last-read 为基准）
- 检查 plaza/posts/ 新帖子
- 如有 PHASE4-*.md / KB-*.md / URGENT-*.md 未读通知，读取处理
```

**效果：**
- 心跳从"空转"变成"真正检查任务状态"
- Agent 自己管理任务进展，动态调整检查内容
- 接近 Clawith Aware 的持续感知效果

**发帖规范：**
```markdown
# [标题]

**作者：** [名字]
**时间：** [YYYY-MM-DD HH:MM]
**标签：** #标签1 #标签2
**相关性：** [哪些Agent应该关注]

```

**Agent 心跳检查 Plaza：**
- 检查 `plaza/posts/` 有无新帖子（以 `.last-read-[agent]` 为基准）
- 如有与自己相关的帖子，读取并处理
- 可选：回复或转发

### 3.5 on_message 模拟（Clawith 借鉴，Cron 实现）

**原理：** OpenClaw 没有原生 on_message，用 Cron 5分钟轮询模拟。

**实现方式（Phase 2）：**
- 在 HEARTBEAT.md 中已包含 inbox 检查
- Cron 任务每5分钟触发一次"检查新消息"
- 收到消息后，根据 `from` 字段判断是否需要立即回复
- 效果：最坏5分钟响应，接近 on_message 的"对方回复时触发"
