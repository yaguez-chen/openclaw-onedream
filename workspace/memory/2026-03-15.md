## 飞书机器人路由问题诊断（2026-03-15 07:48-08:31 GMT+8）

### 问题描述
梦想家发现所有飞书机器人都是"阿尔法"——无论通过哪个机器人发消息，都路由到 main agent（阿尔法）。

### 根因分析
1. **main 绑定使用 `peer.id` 匹配梦想家的 open_id（ou_f276722b...）**
   - 根据文档匹配顺序，`match.peer` 优先级最高（第1级）
   - 所有来自梦想家的消息（无论通过哪个机器人）都先匹配这个 binding
   - 直接路由到 main agent，其他 bindings 永远不会被匹配

2. **其他 bindings 使用了错误的 `accountId` 值**
   - beta: `cli_a939e52002b8dcb0` (App ID 格式，错误)
   - gamma: `ou_01328c78...` (open_id 格式，错误)
   - delta/epsilon/其他: 类似错误
   - 正确的 accountId 应该是账户 key（如 "main", "beta", "gamma"）

3. **部分飞书账号有 WebSocket 连接错误**
   - 400 Bad Request 错误，可能是事件订阅未正确配置

### 解决方案（推荐）
**使用 `accountId` 匹配账户 key**，让每个机器人独立路由到对应 agent：

```json
bindings: [
  { "agentId": "main", "match": { "channel": "feishu", "accountId": "main" } },
  { "agentId": "beta", "match": { "channel": "feishu", "accountId": "beta" } },
  { "agentId": "gamma", "match": { "channel": "feishu", "accountId": "gamma" } },
  // ... 其他机器人类似
]
```

**理由：**
- 文档明确指出 "Use `bindings[].match.accountId` to route each account to a different agent."
- 移除 main 的 `peer.id` 匹配，改为 `accountId: "main"`
- 每个机器人账户对应一个 agent，消息进来时系统识别是哪个账户收到的
- 路由匹配顺序：peer > guildId > teamId > accountId > accountId:* > default

### 预期效果
- 阿尔法机器人 → 路由到 main agent
- 贝塔机器人 → 路由到 beta agent
- 伽马机器人 → 路由到 gamma agent
- ...以此类推

### 待执行
- 修改 openclaw.json 的 bindings 部分
- 重启 gateway
- 测试验证

### 参考
- OpenClaw 配置参考：bindings match fields
- Feishu 文档：Multi-agent routing
- 日志证据：`feishu[beta]: dispatching to agent (session=agent:main:...)`
